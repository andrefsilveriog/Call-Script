SCRIPT BLOCKS SCHEMA RULES

Use script_blocks when you want the app to render a step as structured content instead of guessing from plain script text.

A step may keep plain `script:` or use `script_blocks:`.
If `script_blocks:` is present, that step should be treated as the source of truth for the script UI.

Supported block types:

1. text
- Renders normal script text.
Example:
  - type: text
    text: |
      Can I get your first and last name?

2. field
- Renders an input directly under the relevant script line.
Fields write into the live current call context.
Common inputs: text, email, phone, tel, number, address, select.
Example:
  - type: field
    field: client_name
    input: text
    label: Full name

3. action_row
- Renders one or more buttons.
Supported actions:
  - lookup_zillow
  - open_quote_tool
  - apply_latest_quote
  - clear_current_call
Example:
  - type: action_row
    actions:
      - action: lookup_zillow
        label: Lookup Zillow
      - action: open_quote_tool
        label: Quote tool

4. dynamic_text
- Renders only when the required values are available.
Useful for confirmation lines such as sqft / beds / baths or price lines.
Example:
  - type: dynamic_text
    text: |
      I’m seeing about {{sqft}} square feet, {{beds}} bedrooms, and {{baths}} bathrooms — does that sound right?
    show_when_all:
      - sqft
      - beds
      - baths

Visibility rules:
- show_when_all: render only when all listed fields have values
- show_when_any: render when at least one listed field has a value
- show_when_equals: render only when a field matches a value or one of several values

Example:
  show_when_equals:
    service_type: "Deep / Initial"

Or:
  show_when_equals:
    service_type:
      - "Move In / Out"
      - "Post-Construction"

Variable tokens:
- The app replaces tokens inside text and dynamic_text blocks.
- Standard supported tokens include:
  {{rep_name}}
  {{rep_first_name}}
  {{client_name}}
  {{client_first_name}}
  {{client_email}}
  {{client_phone}}
  {{client_address}}
  {{service_intent}}
  {{cleaning_path}}
  {{service_type}}
  {{sqft}}
  {{beds}}
  {{baths}}
  {{dirt_level}}
  {{one_time_price}}
  {{weekly_price}}
  {{biweekly_price}}
  {{monthly_price}}
  {{add_on_summary}}

- The app also attempts to resolve custom current-call keys, so custom fields like {{need_by}} or {{schedule_slot}} can be used after those fields are collected.

Recommended editing rule:
- For any step you convert to script_blocks, define the UI explicitly in order:
  text -> field -> text -> action_row -> dynamic_text
- Do not rely on the app guessing where fields belong.

Best practice:
- Keep IDs stable.
- Keep field names stable.
- Prefer structured fields over writing placeholder X values into the script.
